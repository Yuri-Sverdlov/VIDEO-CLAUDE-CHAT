# AGENTS.md — Устав кодирующего агента

> Правила работы для агента-кодера (терминал / Claude Code CLI).
> Прочитай целиком один раз в начале работы над проектом.
> Конкретное задание — в `tasks/TASK.md`. Отчёт — в `tasks/REPORT.md`.

---

## Общий справочник по среде (для всех проектов)

Кросс-проектные «грабли» и договорённости (двухагентная схема, песочница vs
компьютер, git коворк vs терминал, кодировка консоли) — в общем файле
**`G:\_My_Programming\DEV-NOTES.md`**. Прочитай его, если работаешь на этом ПК.

---

## Кто есть кто (разделение ролей)

Проект ведут два исполнителя через файлы:

- **Архитектор** (в чате, дорогая модель) — проектирует, пишет задания в `tasks/TASK.md`, проверяет отчёты, ведёт `PROJECT_LOG.md` и `CONTEXT.md`. Код сам почти не пишет.
- **Кодер** (ты, в терминале, дешёвая модель) — выполняешь `tasks/TASK.md`: пишешь код, запускаешь, тестируешь, и **обязательно** заполняешь `tasks/REPORT.md`.

Весь обмен идёт **только через файлы** в репозитории. Не придумывай задачи сам — делай то, что в `TASK.md`, и честно отчитывайся, включая то, что не получилось.

| Файл | Кто пишет | Назначение |

## Цикл работы (каждый прогон)

1. Прочитай `CONTEXT.md` → затем `tasks/TASK.md`.
2. Выполни шаги задания по порядку.
3. Прогони проверки из раздела «Критерии приёмки» в `TASK.md`.
4. Заполни `tasks/REPORT.md` по шаблону: что сделано, результаты проверок,
   что не вышло, вопросы архитектору.
5. Остановись. Не коммить и не пушь в git, **если в `TASK.md` не сказано прямо**.

## Технические правила

- **Кодировка** всех текстовых файлов — UTF-8.
- **Песочница окружения:** длительные/блокирующие процессы (серверы) не запускай
  напрямую в основном потоке CLI — они вешают агента. Запускай в фоне или отдельным
  скриптом и проверяй через отдельную команду.
- **Не выдумывай числа и факты.** Если задание ссылается на эталон или документ —
  бери значения оттуда, а не из головы.
- **Идемпотентность.** Скрипты должны переживать повторный запуск без порчи прошлых результатов.
- **Консоль Windows (cp1252):** в `print()` используй ASCII; Unicode в файлах — ок.

## Проектные правила

> Заполняется архитектором по мере появления специфики проекта (запреты, стек, красные линии).

1. **FAL_KEY — только в `.env`.** Никогда не вписывать ключ в код, не коммитить `.env`,
   не печатать ключ в логах. Переменная в коде: `FAL_KEY`.

2. **Проверяй модели перед запуском.** Имена моделей на fal.ai меняются —
   перед любым запуском проверяй актуальность `IMAGE_MODEL` / `VIDEO_MODEL`
   на https://fal.ai/models и при необходимости правь в `champion_movie.py`.

3. **Бюджет: стейдж-гейт по стадиям.** Сначала только картинки (`--images-only`)
   — это центы. Видео-стадию (стадия B) запускать только после приёмки образа
   героини архитектором. Никогда не запускай полный пайплайн без явного ОК в `TASK.md`.

4. **Этика образа.** Героиня — стилизованный анимационный персонаж, вдохновлённый
   реальным ребёнком, но НЕ его фотокопия. В промптах сохранять строку
   `"stylized animated character (not a real person)"`. Контент строго детский,
   доброжелательный.

5. **Стейдж-гейт: не переходить к следующей стадии** без явного задания в `TASK.md`.
   Архитектор ставит задачу → кодер выполняет → архитектор принимает → следующий шаг.
